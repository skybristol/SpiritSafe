# SpiritSafe

**The Entity Profile Registry for the Global Knowledge Commons**

SpiritSafe is a centralized, version-controlled registry of GKC Entity Profiles—YAML-based definitions that specify how entities are structured, validated, and transformed across the Global Knowledge Commons. Each profile serves as a community-maintained package that includes:

- **Profile Definition** (YAML) — Canonical structure of an entity type
- **Controlled Vocabularies** (SPARQL queries) — Definitive lists of allowed values for properties
- **Metadata & Documentation** — Version history, authorship, and usage guidance
- **Governance** — PR review, CI validation, and community contribution

## Vision

SpiritSafe reimagines entity profiles as **registry entries** with explicit versioning, documentation, and governance—solving the adoption and clarity problems that have plagued Wikidata's EntitySchema (ShEx) system. By treating profiles as community assets with clear ownership and contribution processes, SpiritSafe makes entity schema management transparent, auditable, and accessible.

## What's in the Registry?

This repository contains:

### **Profiles** (`profiles/`)

Entity profile packages, each in its own directory:

- `TribalGovernmentUS/` — Federally recognized Native American tribes
- `OfficeHeldByHeadOfState/` — Governmental executive offices
- _(additional profiles as the registry grows)_

Each profile directory includes:

- `profile.yaml` — The entity profile definition
- `metadata.yaml` — Version, status, authors, and discovery metadata
- `README.md` — Entity description, usage guidance, and examples
- `CHANGELOG.md` — Version history and change tracking
- `queries/` — SPARQL queries for this profile's controlled vocabularies (if applicable)

### **Shared Queries** (`queries/`)

SPARQL queries used across multiple profiles or as common references.

### **Cached Results** (`cache/`)

Cached SPARQL query results (generated by CI workflows) for fast profile loading without live Wikidata queries. Organized to mirror the profile/query structure:

```
cache/
├── profiles/
│   └── TribalGovernmentUS/
│       ├── bia_federal_register_issues.json
│       └── wikidata_language_items_en.json
└── queries/
    └── [shared query caches]
```

### **CI/CD Workflows** (`.github/workflows/`)

- **`validate-profile.yml`** — Validates YAML syntax, metadata, and SPARQL queries on PRs
- **`hydrate-caches.yml`** — Executes SPARQL queries and updates caches (manual, scheduled, or on merge)
- **`publish-release.yml`** — Creates tagged releases when profile versions change

## Getting Started

### Using Profiles

Profiles are designed to be consumed by the **gkc** Python package via the `gkc.spirit_safe` module.

**Example**:

```python
from gkc.spirit_safe import SpiritSafeSourceConfig, set_spirit_safe_source
from gkc.profiles import ProfileLoader

# Configure source (GitHub or local SpiritSafe)
set_spirit_safe_source(
    mode="github",
    github_repo="skybristol/SpiritSafe",
    github_ref="main"
)

# Load profile
loader = ProfileLoader()
profile = loader.load_from_file("profiles/TribalGovernmentUS/profile.yaml")
```

### Contributing a New Profile

1. **Fork** this repository
2. **Create a branch**: `git checkout -b profile/my-new-profile`
3. **Create profile directory**: `profiles/MyNewProfile/`
4. **Add required files**:
   - `profile.yaml` — Entity profile definition
   - `metadata.yaml` — Version, status, authors, etc.
   - `README.md` — Entity description and guidance
   - `CHANGELOG.md` — Version history
   - `queries/` — SPARQL queries (if needed)
5. **Commit and push**: `git push origin profile/my-new-profile`
6. **Create a pull request** with description of the new profile
7. **CI validation** will run automatically
8. **Await maintainer review** and address feedback
9. **Merge** — Profile becomes part of the registry

### Updating an Existing Profile

1. **Fork** and create a branch: `git checkout -b profile/tribal-government-update`
2. **Edit** the profile files in `profiles/[ProfileName]/`
3. **Update** `CHANGELOG.md` with your changes
4. **Increment version** in `metadata.yaml` (follow [semantic versioning](https://semver.org/))
5. **Commit, push, and create a PR**
6. **CI validation** runs; maintainer reviews
7. **Merge** — New version is tagged and released automatically

## Profile Standards

All profiles must include:

- ✅ Valid YAML syntax in `profile.yaml`
- ✅ Complete `metadata.yaml` with all required fields
- ✅ Comprehensive `README.md` documenting the entity type and usage
- ✅ `CHANGELOG.md` tracking all version changes
- ✅ Valid SPARQL syntax in query files (if applicable)

CI workflows enforce these standards on every PR.
